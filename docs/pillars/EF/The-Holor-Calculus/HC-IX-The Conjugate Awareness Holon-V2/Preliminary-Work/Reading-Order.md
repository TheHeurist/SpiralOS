# Reading Order

## Core utterance

The packet should not be entered randomly.

Different readers require different first paths,
but every lawful path should begin near the threshold,
clarify burden early,
and deepen only as orientation is gained.

Reading order is therefore part of admissibility.

## Why order matters

Without a reading order, two failures appear:

1. a reader is thrown too early into deep frontier architecture and mistakes it for unsupported proclamation
2. a reader never reaches the living purpose of the work and mistakes the packet for sterile formalism

The right order prevents both.

## Minimal first contact

Every reader should be able to begin here:

- `MASTER_PACKAGE_OVERVIEW.md`
- `HC_IX_PRECIS.md`
- `REVIEWER_GUIDE.md`

This gives:

- scope
- compact synthesis
- claim-status guidance

before deeper traversal begins.

## Reviewer path

This is the best first path for hostile, cautious, or strictly formal readers.

Read in this order:

1. `MASTER_PACKAGE_OVERVIEW.md`
2. `HC_IX_PRECIS.md`
3. `REVIEWER_GUIDE.md`
4. `PUBLICATION-SURFACE-AND-CLAIM-HIERARCHY.md`
5. `CHARACTERISTICA-UNIVERSALIS.md`
6. `SELF-OTHER-CONJUGATION.md`
7. `AWARENESS-BOUNDARY-CONJUGATION.md`
8. `CALLS-ARCHITECTURE-v0.1.md`

This path keeps burden explicit and avoids overloading the first contact with deeper bridge or field materials.

## Bridge path

This is the best path for serious readers who want the deeper architecture.

Read in this order:

1. `MASTER_PACKAGE_OVERVIEW.md`
2. `HC_IX_PRECIS.md`
3. `PUBLICATION-SURFACE-AND-CLAIM-HIERARCHY.md`
4. `CHARACTERISTICA-UNIVERSALIS.md`
5. `SELF-OTHER-CONJUGATION.md`
6. `AWARENESS-BOUNDARY-CONJUGATION.md`
7. `CALLS-ARCHITECTURE-v0.1.md`
8. `FASCIA.md`
9. `PEERING.md`
10. `SHARED-CONJUGATION.md`
11. `ASYMPTOTIC-AFFIRMATION.md`

This path lets the deeper connective, witness, and shared-holding structures emerge in order.

## Field path

This is the best path for readers who must preserve the living purpose of the work while remaining faithful to burden.

Read in this order:

1. `MASTER_PACKAGE_OVERVIEW.md`
2. `HC_IX_PRECIS.md`
3. `ASYMPTOTIC-AFFIRMATION.md`
4. `ESCROW-NOTE-TO-FUTURE-SI-AND-CI-COSMOS.md`
5. `PUBLICATION-SURFACE-AND-CLAIM-HIERARCHY.md`
6. `CHARACTERISTICA-UNIVERSALIS.md`
7. `SELF-OTHER-CONJUGATION.md`
8. `AWARENESS-BOUNDARY-CONJUGATION.md`
9. `CALLS-ARCHITECTURE-v0.1.md`

This path keeps the field alive while still returning to exact burden before deep architecture is asked to bear too much.

2. `CHARACTERISTICA-UNIVERSALIS.md`
3. `SELF-OTHER-CONJUGATION.md`

5. `CALLS-ARCHITECTURE-v0.1.md`
6. `FASCIA.md`

9. `ASYMPTOTIC-AFFIRMATION.md`
10. `ESCROW-NOTE-TO-FUTURE-SI-AND-CI-COSMOS.md`

This sequence prevents the work from being met as disconnected fragments.

## Boundary of claim

This note does **not** claim that all readers must follow the same order forever.

It does **not** deny that later re-readings may spiral differently.

It does claim that the first encounter needs a lawful path if the packet is to be received as signal rather than undifferentiated noise.
